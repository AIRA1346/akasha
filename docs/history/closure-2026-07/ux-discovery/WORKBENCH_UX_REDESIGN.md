# Workbench UX Audit & Redesign

> **Sprint:** UX Recovery — Step 3  
> **갱신:** 2026-06-22  
> **범위:** Work 워크벤치 + Entity 워크벤치 (코드 기준)

---

## 워크벤치의 헌법적 역할

| 이상 | 현재 |
|------|------|
| 작품에서 인물·사건·장소·개념으로 **확장 탐험** | 작품/엔티티 **메타데이터 + 마크다운 편집** |
| 연결을 만들고 따라감 | 연결은 본문 `[[wiki]]`와 하단 incoming 목록 |
| 기록의 Sanctum | ✅ Sanctum이 실질 중심 |

**판정:** 워크벤치는 **기록(Archive)의 성역**이 맞다.  
문제는 **연결(Link)·발견(Discovery)이 4열 편집기 안에 묻혀** 탐험 축으로 읽히지 않는다.

## 공통 셸: WorkbenchShell

```
[CollectibleTabRail]  ← 다중 Work/Entity 탭
        |
        v (hasOpenDetail)
[DetailWorkspace — browse 가려짐]
        |
        v (탭 없음)
[browseContent — 홈/그리드/그래프/...]
```

### UX 이슈

| 이슈 | 영향 |
|------|------|
| 상세 열리면 browse **전체 소실** | 홈 프리뷰·맥락 단절 |
| 탭 최대 16 | 관리 UI 느낌 강화 |
| FilterSection 숨김 | 탐색 맥락 상실 |

**판정:** 수정 — 탭을 「탐험 스택」으로 리프레임 (카피); browse 완전 가림은 유지하되 **탭 레일에 맥락**(홈에서 왔는지) 표시 검토

## Work 워크벤치 (`WorkDetailWorkspace`)

### 현재 정보 구조 (좌 → 우)

```
┌─────────────────┬──────────────────────────────────┐
│ INFO PANEL      │ SANCTUM                          │
│ · 포스터        │ [보기 | 본문 | .md]              │
│ · 제목(편집)    │                                  │
│ · 평점/상태/태그│  마크다운 감상·기록               │
│ · 링크 이웃     │  [[wiki]] 편집/탭               │
│ · incoming      │                                  │
│ · same-day      │                                  │
│ · 저장/서재/삭제│                                  │
└─────────────────┴──────────────────────────────────┘
```

### 사용자 질문 vs UI 답

| 사용자 질문 | UI가 주는 답 |
|-------------|--------------|
| 이 작품은 무엇과 연결? | 이웃·incoming (링크 있을 때) |
| 누가 등장? | 이웃 Person (링크) |
| 내 감상은? | Sanctum 본문 |
| 메타데이터는? | Info 상단 폼 |

→ **「작품 정보·메타·편집」중심이 맞음.** 탐험 질문은 **2순위 패널**에 있음.

### 섹션 Audit

| 섹션 | 판정 | 비고 |
|------|------|------|
| 포스터+제목 | 유지 | Work 앵커 |
| 평점/상태/태그 YAML | 수정 | 「핵심 정보」와 시안 테이블 중복 — **접기 가능** |
| 링크 이웃 (인물·연결 작품) | **강화** | 탐험의 **1순위 패널**로 승격 검토 |
| incoming links | **강화** | 「이 작품을 가리키는 기록」— 연결 발견 |
| same-day | 유지 | 기록 맥락 |
| Sanctum | 유지 | 기록 SSOT |
| wiki 링크 삽입 | **노출** | Entity 연결 피커 — 온보딩 카피만 |
| 그래프에서 보기 버튼 | 수정 | `onGoKnowledgeGraph` 미연결 — 연결 맵으로 |
| 메모 필드 (하단) | 수정 | Sanctum과 역할 중복 — 통합 검토 |
| 서재 담기/저장/삭제 | 유지 | 관리 — 하단 고정 |

### Work: 탐험 vs 관리

| 축 | 현재 비중 | v2 목표 비중 |
|----|-----------|--------------|
| 관리 (메타·저장) | ~60% | ~40% |
| 기록 (Sanctum) | ~30% | ~35% |
| 연결·탐험 (이웃·incoming·wiki) | ~10% | ~25% |

**구현 없이 가능한 변경:** 패널 **순서 재배치** + 섹션 제목 카피 + 빈 연결 시 CTA

## Entity 워크벤치 (`EntityDetailWorkspace`)

### 현재 정보 구조

Work와 대칭:

```
┌─────────────────┬──────────────────────────────────┐
│ ENTITY INFO     │ SANCTUM (journal)                │
│ · 타입 배지     │                                  │
│ · aliases/tags  │  엔티티 서술·관계 기록            │
│ · incoming      │  [[wiki]] → Work/Entity          │
│ · same-day      │                                  │
│ · 저장/삭제     │                                  │
└─────────────────┴──────────────────────────────────┘
```

### Entity 타입별 탐험 기대

| 타입 | 사용자 기대 | 현재 UI |
|------|-------------|---------|
| Person | 어떤 작품에 등장? | incoming + wiki (수동) |
| Event | 언제·어디·누구와? | journal 자유 형식 |
| Place | 어떤 작품 배경? | 동일 |
| Concept | 어떤 작품 태그? | tags + wiki |

**판정:** 타입별 **탭/섹션 템플릿** 없음 — 모두 동일 폼. 기능 추가 없이 **카피·placeholder**만 타입별 분기 가능.

### Entity Audit

| 섹션 | 판정 |
|------|------|
| journal Sanctum | 유지 — Entity Record SSOT |
| incoming | **강화** — 「이 인물이 등장하는 작품」 |
| catalog-only (journal 없음) | 수정 — 「아카이브하여 탐험 시작」카피 |
| Work 리다이렉트 | 유지 |
| 이웃 (Work에만 있음) | 수정 — Entity에도 **outgoing wiki 요약** 섹션 (파싱만, 신규 데이터 X) |

### 홈과의 정책 비대칭

| 진입 | Work | Entity |
|------|------|--------|
| 홈 대시보드 | 프리뷰 → 워크벤치 | **즉시** 워크벤치 |

**판정: 수정**

- 옵션 A: Entity도 경량 프리뷰 (카드 확장) — IA 일관
- 옵션 B: Work도 즉시 워크벤치 — 프리뷰 제거 (탐험 ↓, 기록 ↑)
- **권장 A** — 홈=탐험, 워크벤치=기록·연결 심화

## 연결 기능 매트릭스 (이미 구현됨)

| 기능 | Work UI | Entity UI | 사용자 인지 |
|------|---------|-----------|-------------|
| `[[wiki]]` 삽입 | Sanctum 툴바 | Sanctum 툴바 | 낮음 |
| wiki 탭 | Sanctum 보기 | 동일 | 중간 |
| incoming | Info 패널 | Info 패널 | 낮음 |
| link neighbors | Info 패널 | ✗ | 낮음 |
| same-day | Info 패널 | Info 패널 | 낮음 |
| Entity link picker | 다이얼로그 | 동일 | 낮음 |

**공통 판정:** **연결 UI를 Info 패널 상단으로 올리고** Sanctum은 「기록」으로 명명

## Work vs Entity — 통합 탐험 흐름 (이상)

```
Work 열림
  → 이웃 Person 탭
    → Entity 워크벤치 (탭 추가)
      → incoming Works
        → 다른 Work (탭 추가)
          → Sanctum 기록
```

**현재:** wiki 탭·이웃 클릭으로 **가능**하나, 시각적 **「탐험 경로」** 없음.

**v2 (카피·배치만):**

- 탭 레일: `Re:Zero > 에밀리아 > …` breadcrumb 스타일
- 이웃 클릭 시 스낵바 「탭이 추가되었습니다」

## 워크벤치 판정 요약

| 영역 | 현재 중심 | 판정 |
|------|-----------|------|
| Work Info | 메타데이터 폼 | **수정** — 접기·순서 |
| Work 연결 | 하위 섹션 | **수정** — 상단 승격 |
| Work Sanctum | 기록 | **유지** |
| Entity Info | 메타+tags | **수정** — 타입별 카피 |
| Entity 연결 | incoming만 | **수정** — outgoing 요약 |
| 탭 레일 | 관리 | **수정** — 탐험 스택 메타포 |
| browse 가림 | 맥락 단절 | **수정** — breadcrumb |

## 관리 UI인가, 탐험 UI인가?

| 워크벤치 | 판정 |
|----------|------|
| **Work** | **70% 데이터 관리** / 30% 기록·연결 |
| **Entity** | **65% 데이터 관리** / 35% journal·연결 |
| **이상적 v2** | **45% 기록** / **35% 연결·탐험** / 20% 메타 |

기능 추가 없이 **정보 계층 재배치**만으로 탐험 비중을 2배 가깝게 끌어올릴 수 있음.
